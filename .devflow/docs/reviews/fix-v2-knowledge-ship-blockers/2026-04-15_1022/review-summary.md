### HIGH (8 issues)

**1. Shell-substitution placeholder leaks command injection across 8 orchestration surfaces** — Confidence: 88%
- **Locations**: 8 surfaces (resolve.md/.md-teams, code-review.md/.md-teams, plan.md/.md-teams, self-review.md, plus 4 orch skills)
- **Issue**: Orchestrator substitutes `<worktree>` / `{worktree}` without quoting normalization into double-quoted shell argument. If path contains `"` or `$(...)` or backticks, commands execute.
- **Impact**: Command injection via crafted worktree path (typical user input).
- **Fix**: Use `printf %q` or pin to cwd literal `"."` with `git -C` pattern; validate paths in worktree-support skill.

**2. Inconsistent worktree placeholder convention (3 distinct forms)** — Confidence: 96%
- **Locations**: resolve.md/resolve-teams.md use `"<worktree>"`, code-review.md/code-review-teams.md use `"{worktree}"`, plan.md/plan-teams.md/orch-skills use `"."`
- **Issue**: Three competing conventions for the same operation across 8 surfaces. Harder to substitute correctly.
- **Impact**: Silent knowledge loss if wrong form substituted into path.resolve().
- **Fix**: Standardize on `{worktree}` for command surfaces and `"."` for orch skills; tighten tests.

**3. `/debug` and `/debug-teams` left on old "fan-out corpus" pattern while orch skill and CHANGELOG claim PF-011 closed** — Confidence: 90%
- **Locations**: `plugins/devflow-debug/commands/debug.md:36`, `debug-teams.md:28`, vs `shared/skills/debug:orch/SKILL.md` (updated)
- **Issue**: Documentation drift. CHANGELOG line 17 claims "closes PF-011 across all four knowledge-consuming commands" but `/debug` surfaces still read full corpus. The orch skill was updated; the slash commands were not.
- **Impact**: Users running `/debug` get old behavior while code-review/plan/resolve get new behavior — inconsistent handling of PF-011.
- **Fix**: Either update debug.md/debug-teams.md to use index pattern matching debug:orch, OR scope CHANGELOG claim to `:orch` skills only with a note.

**4. Inconsistent `KNOWLEDGE_CONTEXT` placeholder quoting across 11+ prompt-template sites** — Confidence: 92%
- **Locations**: 11 sites using `{knowledge_context or '(none)'}` vs `{knowledge index from Step 0d, or (none)}` vs `{Phase 2 knowledge index, or (none)}` vs various quote styles
- **Issue**: Quoting inconsistency causes `apply-knowledge` skip-guard to silently fail. The guard checks for unquoted `(none)`, but some sites substitute `'(none)'` with quotes.
- **Impact**: Reviewer silently receives quoted literal instead of skipping, no reads performed, no work done.
- **Fix**: Standardize on `{knowledge_context or (none)}` (no quotes) across all surfaces. Document the literal token is `(none)`.

**5. `/implement` and `pipeline:orch` bypass new knowledge pattern despite agents declaring `devflow:apply-knowledge`** — Confidence: 92%
- **Locations**: `plugins/devflow-implement/commands/implement.md`, `shared/skills/implement:orch/SKILL.md:91-93`, `pipeline:orch/SKILL.md`
- **Issue**: Simplifier and Scrutinizer agents declare `KNOWLEDGE_CONTEXT` input and Apply Knowledge section, but orchestrators don't load or pass knowledge. Agents fall through skip-guard.
- **Impact**: Implementation workflow (highest-leverage place for pitfall awareness) never sees knowledge context.
- **Fix**: Either extend Phase 0 knowledge load to `/implement` + thread `KNOWLEDGE_CONTEXT` to Coder/Simplifier/Scrutinizer, OR explicitly document v2 scope as {resolve, plan, self-review, code-review} only with rationale.

**6. Layering inversion: `apply-knowledge` skill hardcodes paths, contradicts index footer's single-source-of-truth contract** — Confidence: 86%
- **Locations**: `shared/skills/apply-knowledge/SKILL.md:36-38, 54-58`
- **Issue**: Skill body says "Read `.memory/knowledge/decisions.md`" with hardcoded paths. Footer from index says "ADR-NNN entries live in {path}" (computed from worktree). Two sources of truth.
- **Impact**: Consumer agent in multi-worktree mode with `WORKTREE_PATH` set reads wrong directory.
- **Fix**: Remove hardcoded paths. Replace with: "path is in index footer — Read that file."

**7. Inconsistent `KNOWLEDGE_CONTEXT` placeholder terminology in agent input docs** — Confidence: 92%
- **Locations**: code-review.md uses `{knowledge_context or '(none)'}`, resolve.md uses `{knowledge index from Step 0d, or (none)}`, plan.md uses different phrasing
- **Issue**: Same input variable described with different names/structures across 5+ surfaces.
- **Impact**: Orchestrator substitution ambiguity; unclear whether `knowledge_context` or `knowledge_index` is the right variable name.
- **Fix**: Standardize variable name and description shape across all 11+ prompt-template sites.

**8. `legacy-knowledge-purge.ts:212-217` unconditionally `fs.unlink`s without symlink rejection** — Confidence: 82%
- **Location**: `src/cli/utils/legacy-knowledge-purge.ts:212-217`
- **Issue**: Calls `fs.unlink()` on `.memory/PROJECT-PATTERNS.md` without `lstat` + symlink check. If path is a symlink (attacker-controlled or accidental), deletes the target.
- **Impact**: Low likelihood but inconsistent with codebase's explicit symlink defense on writes (D39 in fs-atomic.ts).
- **Fix**: Add `lstat` + `S_ISREG` check before unlink.
