### Q4: chmod guard simplification

**Verdict**: No regression. **Confidence**: 98%

The old one-liner:
```bash
[ ! -f "$QUEUE_FILE" ] && touch "$QUEUE_FILE" && chmod 600 "$QUEUE_FILE" 2>/dev/null || true
```

The new if-block:
```bash
if [ ! -f "$QUEUE_FILE" ]; then
  touch "$QUEUE_FILE" && chmod 600 "$QUEUE_FILE" 2>/dev/null || true
fi
```

These are semantically identical. The `&&`/`||` chain in the one-liner has the same control flow as the if-block version: when the file exists, skip; when it does not, touch+chmod with fallback. The if-block is clearer about the intent -- the `|| true` now unambiguously applies only to the touch/chmod failure case, not to the file-exists case.

**Verdict**: No regression, defensive improvement. **Confidence**: 95%

Old: `grep -q '"role":"assistant"'` (matches compact JSON only)
New: `grep -q '"role"[[:space:]]*:[[:space:]]*"assistant"'` (matches any whitespace around colon)

Both `jq -c` and `JSON.stringify` produce compact JSON (no spaces), so the old pattern would still work for queue files generated by this hook. The widened pattern is defensive against any future change to JSON formatting or manual queue edits. No false positives possible -- the pattern is strict on the key and value, only flexible on whitespace.
