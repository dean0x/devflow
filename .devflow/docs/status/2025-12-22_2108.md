## Architecture & Design

### Key Architectural Decisions

**Pure Orchestration Pattern:**
```
Command (orchestrator)
├─ Spawns agents in parallel where possible
├─ Passes structured context to each agent
├─ Waits for agent completion
└─ Displays agent outputs (no modification)
```

**Agent Responsibilities:**
- Review agents → Find issues, create reports
- Comment agent → Create PR inline comments
- TechDebt agent → Track pre-existing issues in GitHub issue
- Summary agent → Aggregate findings, make recommendation
- Synthesize agent → Combine parallel agent outputs

**Parallel Execution:**
- Phase 3 (Review): 6+ review agents run in parallel
- Phase 4 (Synthesis): Comment + TechDebt + Summary run in parallel
- Swarm: 4 explorers parallel, 3 planners parallel, N coders parallel

### Technology Stack

- **CLI**: TypeScript, Node.js
- **Commands/Agents**: Markdown with YAML frontmatter
- **Version Control**: Git with worktree support for parallel implementation
- **GitHub Integration**: gh CLI for PRs, comments, issues

### Design Patterns in Use

1. **Orchestration over delegation** - Commands coordinate, agents execute
2. **Parallel-first** - Independent work always parallelized
3. **Single responsibility** - Each agent does one thing well
4. **Structured handoffs** - Clear data contracts between phases

## Context for Future Developer

### Things to Know

1. **$ARGUMENTS is how commands get input** - Whatever follows `/command` becomes `$ARGUMENTS`
2. **Skills install flat** - Under `~/.claude/skills/` directly, not in subdirectory
3. **TypeScript compiles differently** - `src/cli/commands/` becomes `dist/commands/` (rootDir strips `cli/`)
